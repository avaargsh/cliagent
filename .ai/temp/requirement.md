# Python 数字员工控制面产品需求（P1）

## 0. 假设声明

当前未收到额外业务输入，本需求基于以下假设（待产品负责人确认）：

1. 首发版本目标是“控制面 CLI 可用”，不是 Web 控制台或完整开放 API。
2. 以 `work-order` 为一等对象，优先保证“可审计、可恢复、可回放”。
3. `mock` provider 是默认执行路径；`openai` 以配置可接入和自动化回归通过为准。
4. 首发支持 `darwin/linux`，暂不承诺 Windows。

## 1. MVP 摘要

交付一个可脚本化的数字员工控制面 CLI（`dectl`），支持工作单创建/执行、审批、会话观察、事件回放与基础诊断，并提供稳定机读输出（`--json/--jsonl`）与可验证错误协议（exit code + 结构化错误）。

## 2. 目标用户与典型使用场景

- 目标用户：
  - 平台管理员
  - 集成开发者
  - QA / 运维工程师
  - 需要审核高风险动作的业务管理员
- 典型场景：
  - 提交业务任务并跟踪执行状态
  - 高风险工具调用审批与恢复执行
  - 排查后台执行异常与陈旧任务
  - 按工单回放事件链路进行复盘

## 3. 核心用户故事

1. 作为管理员，我希望创建并运行工作单，以便把任务交给数字员工处理。
2. 作为审批人，我希望查看并决策待审批项，并可一键恢复执行。
3. 作为运维，我希望观察 session/work-order 事件流并定位故障。
4. 作为 QA，我希望回放工单事件时间线并验证执行链路。
5. 作为集成开发者，我希望所有成功/失败输出都可机读解析。

## 4. 验收标准（MVP）

- [ ] `work-order create|get|list|run|resume|cancel|watch|artifacts|reclaim` 可执行
- [ ] `approval list|get|decide` 可执行，`--resume --background` 可用
- [ ] `session list|get|tail|export` 可执行，`tail/watch` 支持 `--jsonl --follow`
- [ ] `replay run` 可返回结构化事件时间线
- [ ] `doctor` 可诊断 stale background session
- [ ] `--json` 输出稳定为单对象，错误输出包含稳定 `error.type` 与 exit code
- [ ] 文件状态目录按租户隔离
- [ ] 全量回归测试通过（当前基线 `python3 -m unittest discover -s tests -p 'test_*.py'`）

## 5. 功能需求

1. 命令入口：
  - 提供 `config/employee/work-order/approval/session/tool/replay/doctor/version` 一级命令。
2. 工作单主链：
  - 支持创建、前后台执行、审批暂停、恢复、取消、回收、交付物查询。
3. 审批与策略：
  - 高风险工具动作进入审批；审批后可恢复执行。
4. 会话与观测：
  - 记录 session + events，支持查询、导出、流式观察。
5. 回放能力：
  - 基于事件账本按工单回放执行时间线。
6. 诊断能力：
  - 检测后台心跳异常和陈旧运行。
7. 多租户隔离：
  - 状态目录按租户分目录，跨租户不可见。

## 6. 非功能需求（可量化）

1. 可测试性：
  - 全量回归测试必须可在本地命令一次跑通；当前目标 `>=98` 用例通过。
2. 协议稳定性：
  - `--json` 输出保持 `schema_version=1` 包裹结构稳定。
3. 可观测性：
  - `session tail/work-order watch --jsonl` 每行输出完整 JSON 对象。
4. 并发一致性：
  - 文件写入采用锁 + 原子替换，避免半写状态被读取。
5. 安全性：
  - 不在日志/状态文件输出明文密钥。

## 7. 范围内 / 范围外

- 范围内：
  - CLI 控制面能力闭环
  - `mock/openai` provider 接入
  - 文件仓储 + 事件账本 + 投影
  - 背景执行和恢复链路
- 范围外：
  - REST 端到端可用性承诺
  - Windows 平台支持
  - 数据库持久化替代文件仓储
  - `--profile` 分层配置落地

## 8. 风险与待确认问题

1. 文件仓储扩展性风险：
  - 事件量增长后 `events.jsonl` 全量读取成本上升。
2. 运行时取消语义风险：
  - 部分外部调用仍是线程封装的阻塞 I/O，严格可取消能力待后续增强。
3. 待确认问题：
  - 是否将“全局参数必须前置”作为长期 CLI 约束，还是后续兼容后置写法。
  - 是否在 `0.1.x` 周期内引入真实 OpenAI 联调验收门槛。
